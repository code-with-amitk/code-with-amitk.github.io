What is DRL(distributed rate limiter)

DRL decides whether a client (user, IP, or API key) will proceed to origin/application server, using rate limiting counters. Eg: 100 HTTP req/sec. if 101 arrives with in second, 429(too many requests) should be returned to client

Requirements

1. DRL run in multiple regions Since DRL runs in several regions(US, India etc) atomic counters to be shared across regions so the limit is enforced for the fleet — not per machine.
2. System must be scalable: Adding rate-limiter nodes should not crash the system.
3. Thread-safe: Concurrent checks from many threads/pods must update counters atomically so two simultaneous requests cannot both pass when only one token remains.
4. Low-latency: Rate limiting counters are checked hot path so least latency.

High Level Design

Where the limiter sits: Global LB → Regional LB → AZ LB → Rate Limiter

Cloud topology stacks as Region → Availability Zones (AZs) → data centers. Traffic passes through three load-balancing layers before it reaches the distributed rate limiter. The global load balancer (DNS, geo-DNS, or anycast) receives customer traffic and picks the nearest healthy region — for example sending a Mumbai customer toward ap-south-1 and a New York customer toward us-east-1. Inside that region a regional load balancer spreads connections across AZs in the region. Inside each AZ an availability-zone load balancer picks a healthy gateway pod in that AZ. Only after this third hop does the request hit the API Gateway + distributed rate limiter, which runs the token-bucket check against regional Redis and then forwards allowed calls to backend services.

The limiter is therefore not a single box at the global edge, the region edge, or the AZ edge. It is a fleet of stateless gateway pods deployed in every AZ, each performing the same check against the same regional Redis cluster (replicated multi-AZ inside the region for HA). Global and regional LBs only route; they do not hold token counters. The AZ LB only picks a pod; the pod owns the application-level decision.

For a SaaS provider such as Netscope, inbound traffic is not anonymous web browsing — it is authenticated API traffic from enterprise customers using Netscope services. That distinction matters for placement: the global LB sees hostnames, IPs, and TLS handshakes, but the rate-limit key is usually an API key, tenant id, or OAuth subject extracted only after TLS termination and gateway parsing inside the chosen region. The distributed rate limiter belongs at that application edge, immediately after the AZ LB has delivered the request to a gateway pod.

Why the limiter does not sit on the Global LB

Putting distributed rate limiting on the global load balancer is tempting because that is the first Netscope-owned hop after the customer. In practice the global tier is the wrong place for per-customer token buckets.

The global LB’s job is geo routing and health: steer traffic to a region, fail over when a region is down. It is typically implemented as DNS latency routing, anycast to regional PoPs, or a thin GSLB — not as a full programmable HTTP proxy fleet that can run Lua scripts and maintain millions of live counters. Even when a CDN or WAF sits in front, that layer enforces coarse IP-level or volumetric rules; it does not replace per-tenant API quotas backed by shared atomic state.

A token-bucket check needs a nearby Redis round-trip on every request (or a local lease refreshed from Redis). If the check ran at the global tier, every API call from India and the US would converge on one global Redis or on cross-region writes. That adds 100–300 ms RTT, turns Redis into a worldwide single point of failure, and creates race conditions when two regions increment the same counter concurrently. Rate limiting is latency-sensitive; the global hop is latency-insensitive routing. Keeping counters regional keeps the hot path inside one AZ-to-Redis RTT (sub‑millisecond to a few ms).

Identity also arrives too late at the global tier. Netscope customers present X-API-Key, JWT bearer tokens, or mTLS client certs. The global LB may terminate TLS at a regional PoP, but authoritative authentication, tenant lookup, and route selection happen on the API Gateway after regional and AZ load balancing. Without that identity the limiter cannot choose the correct bucket (tenant_acme vs tenant_globex). Moving rate limiting to the gateway — after AZ LB — keeps routing dumb and policy smart.

A lightweight global guard can still exist above regions: block obvious DDoS by IP, cap connections per PoP, or apply a very coarse per-ASN ceiling. That is abuse protection, not the distributed per-API-key limiter described here. The fine-grained, thread-safe, scalable limiter runs regionally on the gateway fleet after the AZ load balancer.

The API Gateway remains the natural enforcement point. It terminates TLS (or receives it from an upstream PoP), authenticates the Netscope customer, loads the tenant’s rule from config, calls regional Redis atomically, returns 429 or forwards to the correct backend microservice. A dedicated rate-limit sidecar is optional; the gateway is still the layer that orchestrates the check.

Request flow across layers

A Netscope customer in Mumbai sends an authenticated API call. The global load balancer resolves the hostname and routes the connection to the India region — it does not decrement any token. The regional load balancer in India chooses an AZ (for example ap-south-1a or ap-south-1b). The AZ load balancer inside that zone selects a healthy API Gateway pod. That pod terminates TLS if needed, validates the API key, fetches the tenant rule (100 req/min, burst 20) from cache or global config, and runs the token-bucket Lua script against India regional Redis. If tokens remain, the gateway forwards to the correct Netscope microservice and returns the response; if not, it responds with 429 Too Many Requests and the backend is never contacted. A US customer follows the same ladder inside the US region with US regional Redis. Global and regional LBs add no Redis calls; only the gateway on the hot path touches regional counters.

Low Level Design

Algorithm: token bucket (preferred)

Common algorithms are fixed/sliding window counters, leaky bucket, and token bucket. Fixed windows are simple but allow boundary bursts (2× limit at the minute edge). Sliding windows are fairer but need more state or approximations. Leaky bucket smooths egress to a constant drain rate and is excellent for shaping outbound queues, but it is awkward for “allow this HTTP request now or return 429” because it tends to queue rather than admit bursts. Token bucket is the best default for an API edge: tokens refill at a steady rate, a capacity (burst) allows short spikes, and each request consumes one token — if none remain, deny immediately. That matches how clients and gateways behave.

The algorithm runs in two cooperating places. The authoritative step is an atomic Lua script on Redis (or equivalent Redis module): read tokens and last_refill, refill based on elapsed time, clamp to capacity, try to consume one token, write back, return allow/deny. That keeps the check thread-safe across all gateway pods in the region. Optionally each gateway keeps a local lease of a few tokens in memory so most requests skip Redis; when the lease is empty the pod reserves another small batch from Redis. The lease is an optimization for QPS; correctness still comes from Redis atomicity. 


# Token bucket state (per rule + subject key) — stored in Redis HASH
bucket:{rule_id}:{subject} → {
  tokens:      FLOAT,     # current tokens
  last_refill: TIMESTAMP, # last refill (ms)
}

# Rule parameters (often from config; may be cached on gateway)
rule:{rule_id} → {
  rate:     FLOAT,   # tokens added per second (e.g. 100/60 ≈ 1.67)
  capacity: INT,     # burst size (max tokens)
  key_by:   "api_key" | "user_id" | "ip" | "user_id+route"
}

Regional Redis vs Global Redis

The design uses two Redis tiers with different jobs. They are not duplicates of the same data; splitting them keeps the request path fast and the policy layer consistent worldwide.

Regional Redis holds the live token-bucket state for traffic handled in that region. Every allowed or denied API call updates keys such as bucket:api:tenant_acme with fields tokens and last_refill through an atomic Lua script. All gateway pods in the US region read and write the US regional cluster; all pods in India use the India cluster. Sharding inside the cluster is by hash(rule_id + tenant). This store is optimized for high write QPS, sub‑ms latency, and thread-safe increments — it is the heart of the distributed rate limiter. Counters are not replicated to other regions on each request; that would defeat the purpose of regional placement.

Global Redis (often paired with a config DB or control plane) holds slow-changing policy and identity metadata: which tenants exist, their contracted limits (100 req/min vs 10,000 req/min), burst size, which header defines the subject, per-route overrides, blocklists, and feature flags. Gateways pull or subscribe to these rules at startup and on change; they cache them in memory so the hot path rarely calls global Redis. Global Redis may also store optional aggregate counters for billing or offline quota reconciliation, but not per-request token decrements across continents. When an admin changes a tenant’s limit in the Netscope console, the update lands in global Redis/config and fans out to all regional gateways; existing regional bucket keys can be reset or left to drain naturally. 


# REGIONAL Redis (US cluster) — updated on every API request
bucket:api:tenant_acme     → { tokens: 17.3, last_refill: 1710000000123 }
bucket:api:tenant_globex   → { tokens: 0.0,  last_refill: 1710000000456 }

# GLOBAL Redis — updated when policy changes, not per HTTP call
rule:api_default           → { rate: 1.67/s, capacity: 20, key_by: "api_key" }
rule:tenant_acme           → { rate: 100/min, capacity: 50, plan: "enterprise" }
tenant:tenant_acme         → { api_key_hash: "…", status: "active", home_region: "us" }

In short: regional Redis answers “may this request proceed right now?” with atomic counters close to the gateway. Global Redis answers “what is this tenant allowed under their contract?” and keeps that answer consistent in every region. Splitting policy (global) from counters (regional) is why the limiter lives after the AZ load balancer with regional Redis, while the global load balancer stays a routing-only tier that never touches token buckets.

How the two tiers behave across US and India

Each region runs its own regional Redis Cluster, stretched across AZs for failover. Global Redis is typically deployed in one primary region with read replicas or active-active config replication so gateways in US and India see the same rule set within seconds. A tenant pinned to India by the global LB accrues and spends tokens only in India regional Redis. The same tenant hitting the US PoP accrues a separate regional counter unless sticky routing sends them to one home region. For a hard worldwide ceiling, operators either split quota per region (500 + 500 of 1000), route that tenant to a single home region, or run an async aggregator on global Redis — never a synchronous cross-region INCR on every API call.

Inside one region, thread-safety comes from Lua atomicity on the correct Redis shard; scalability comes from horizontal gateway pods behind AZ LBs plus sharded regional Redis. The API Gateway orchestrates TLS, auth, rule lookup (global/cache), counter update (regional), and routing to Netscope backends — whether the Lua call is in-process or via a sidecar, the placement after the AZ load balancer stays the same.
